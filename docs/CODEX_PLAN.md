# Codex 开发计划

本文档用于指导 Codex 按阶段实施项目。Codex 在每个阶段开始前，应先阅读：

1. `README.md`
2. `docs/PRD.md`
3. 本文档
4. 仓库中后续新增的架构决策与 issue

## 一、开发原则

1. 不直接连接或修改生产数据库，除非已提供明确的只读连接方式和字段白名单。
2. 不猜测现有技术栈、数据库字段和部署环境。
3. 涉及生产数据时，先用脱敏测试数据完成本地闭环。
4. 所有数据库变更必须通过迁移文件完成。
5. 所有外部模型调用必须经过统一适配层，不允许散落在业务代码中。
6. 所有生成任务必须具备状态、重试、错误日志和成本记录。
7. 真实事实、文学改编和虚构内容必须在数据层明确标记。
8. 第一阶段不实现自动发布、图片生成或视频生成。
9. 每个阶段完成后更新本文档、README 或 issue，记录实际实现与偏差。
10. 优先交付可运行的纵向闭环，不一次性搭建复杂多 Agent 框架。

## 二、推荐仓库结构

最终结构需要根据现有技术栈调整。若本仓库是全新项目，可参考：

```text
ai_cat_create/
  apps/
    admin-web/              # 内容运营后台
    api/                    # HTTP API
    worker/                 # 同步与 AI 异步任务
    mcp-server/             # 后续阶段启用
  packages/
    domain/                 # 领域模型与业务规则
    db/                     # schema、迁移、repository
    ai-provider/            # 模型供应商适配层
    prompts/                # 版本化提示词
    shared/                 # 通用类型、日志、鉴权
  docs/
    PRD.md
    CODEX_PLAN.md
    architecture/
  tests/
    fixtures/               # 脱敏测试数据
  .env.example
  docker-compose.yml
  README.md
```

如果现有后台已有成熟技术栈，则优先复用，不要为了符合此结构重写系统。

## 三、Phase 0：仓库与技术摸底

### 目标

让 Codex 在不写核心业务代码前，先确认真实技术环境并产出实施设计。

### 任务

- [ ] 检查仓库现有文件与代码结构；
- [ ] 记录前端、后端、数据库、任务队列、对象存储和部署方式；
- [ ] 确认该仓库是独立新服务还是现有系统的扩展；
- [ ] 新建 `docs/architecture/SYSTEM_CONTEXT.md`；
- [ ] 新建 `docs/architecture/DATA_MAPPING.md`；
- [ ] 提供生产数据字段白名单草案；
- [ ] 提供敏感字段排除清单；
- [ ] 提供开发、测试、生产环境隔离方案；
- [ ] 提供模型供应商适配方案；
- [ ] 把所有未确认信息列为明确 TODO，不得猜测。

### 验收

- 可以明确说明数据如何从生产系统进入 AI 资产库；
- 可以明确证明 AI 服务无法写生产库；
- 所有关键技术选择都有文档记录。

## 四、Phase 1：AI 数据库与只读同步

### 目标

建立独立的 AI 内容资产库，并使用脱敏数据跑通同步。

### 首批数据表

- `source_sync_jobs`
- `source_cat_snapshots`
- `cat_assets`
- `cat_media_assets`
- `audit_logs`

### 任务

- [ ] 建立数据库 schema 与迁移；
- [ ] 定义生产数据到 AI 资产的映射 DTO；
- [ ] 实现字段白名单；
- [ ] 实现脱敏转换；
- [ ] 实现全量同步；
- [ ] 实现基于更新时间的增量同步；
- [ ] 实现单只猫重新同步；
- [ ] 实现幂等与源数据哈希；
- [ ] 实现同步批次日志；
- [ ] 实现失败重试；
- [ ] 为同步流程编写单元测试与集成测试；
- [ ] 提供脱敏 fixture，禁止提交真实隐私数据；
- [ ] 提供最小猫咪资产列表 API。

### 安全约束

- 生产库连接用户必须只具备 SELECT 权限；
- AI 服务配置中不得出现生产写账号；
- 同步模块与内容生成模块使用不同服务账户；
- 日志不得输出完整连接字符串或敏感数据。

### 验收

- 重复执行同一同步任务不产生重复数据；
- 删除或下架状态能够正确同步；
- 敏感字段不会出现在快照、资产和日志中；
- 能通过 API 查询 10 只测试猫咪资产。

## 五、Phase 2：角色卡生成

### 目标

将猫咪资料转换成稳定、结构化、可版本化的角色卡。

### 新增数据表

- `cat_character_profiles`
- `prompt_templates`
- `ai_generation_runs`

### 任务

- [ ] 建立 AI Provider 接口；
- [ ] 至少实现一个文本模型供应商适配器；
- [ ] 支持结构化 JSON 输出校验；
- [ ] 建立角色卡 JSON Schema；
- [ ] 建立角色卡提示词模板与版本号；
- [ ] 建立资料压缩与事实摘要步骤；
- [ ] 生成角色卡；
- [ ] 保存输入来源 ID 和事实依据；
- [ ] 支持角色卡重生成；
- [ ] 支持角色卡人工编辑；
- [ ] 支持 `generated / needs_review / approved / superseded` 状态；
- [ ] 记录模型、token、费用、耗时和错误；
- [ ] 为结构化输出错误、超时和重试编写测试。

### 角色卡结构最低要求

```json
{
  "catId": "string",
  "name": "string",
  "adoptionStatus": "string",
  "publicFacts": [],
  "visualTraits": [],
  "personalityTraits": [
    {"trait": "string", "evidenceSourceIds": []}
  ],
  "realStorySummary": "string",
  "fictionDirections": [],
  "storySeeds": [],
  "forbiddenClaims": [],
  "privateOrRestricted": [],
  "uncertainFacts": [],
  "version": 1
}
```

### 验收

- 10 只测试猫均能生成符合 Schema 的角色卡；
- 每个性格或事实主张可以追溯来源；
- 角色卡可人工修改并保留历史版本；
- 失败任务可重试且不会产生混乱版本。

## 六、Phase 3：内容任务与多平台草稿

### 目标

支持运营人员创建任务，并输出公众号、小红书、抖音/视频号脚本。

### 新增数据表

- `content_tasks`
- `content_drafts`

### 任务

- [ ] 创建内容任务 API；
- [ ] 支持人工选猫；
- [ ] 支持规则推荐猫咪；
- [ ] 推荐结果展示评分理由；
- [ ] 支持内容类型与事实等级；
- [ ] 支持多平台选择；
- [ ] 生成共享故事核心；
- [ ] 基于故事核心生成不同平台版本；
- [ ] 公众号输出标题、导语、正文、行动引导；
- [ ] 小红书输出标题、正文、标签建议；
- [ ] 视频号/抖音输出时长、分镜、旁白、字幕、素材清单和封面文案；
- [ ] 支持局部重写；
- [ ] 支持多方案生成；
- [ ] 建立任务状态机；
- [ ] 保存提示词版本和角色卡版本。

### 选猫 MVP 评分建议

```text
score =
  adoption_status_weight
  + profile_completeness_weight
  + media_quality_weight
  + days_since_last_exposure_weight
  + waiting_duration_weight
  + new_event_weight
  + manual_priority_weight
  - recent_content_duplication_penalty
```

评分权重应配置化，不写死在页面中。

### 验收

- 可为指定猫咪创建多平台内容任务；
- 同一任务的平台版本事实一致；
- 虚构内容有明显栏目与事实等级标识；
- 局部重写不会重新消耗整篇生成成本。

## 七、Phase 4：自动质检

### 目标

在人工审核前自动发现事实、隐私、格式和重复问题。

### 新增数据表

- `content_quality_checks`

### 任务

- [ ] 建立确定性规则检查；
- [ ] 建立模型语义检查；
- [ ] 对照角色卡和资产事实；
- [ ] 检查隐私与联系方式；
- [ ] 检查真实、改编、虚构边界；
- [ ] 检查医疗和领养高风险表述；
- [ ] 检查平台长度与结构；
- [ ] 检查近期内容相似度；
- [ ] 输出 `pass / warning / blocked`；
- [ ] 阻断草稿不能直接通过；
- [ ] 保存规则版本与问题定位。

### 必测错误样例

- 把公猫写成母猫；
- 虚构已经被领养；
- 虚构严重疾病；
- 输出救助人手机号；
- 把“帮猫宇宙”虚构旅行写成真实新闻；
- 同一猫咪连续生成高度重复内容；
- 视频脚本时长明显超限。

### 验收

- 上述预设错误均能被警告或阻断；
- 质检结果能够定位到具体问题与依据；
- 人工可以确认警告，但不能绕过阻断而不留记录。

## 十、Phase 7：内部 MCP

此阶段必须在文本内容闭环稳定后开始。

### 目标

将已有领域服务包装为 MCP，而不是让 MCP 直接查询数据库。

### 首批工具

- [ ] `search_cats`
- [ ] `get_cat_profile`
- [ ] `get_cat_public_facts`
- [ ] `list_cat_media`
- [ ] `get_story_seeds`
- [ ] `create_content_task`
- [ ] `get_content_task`
- [ ] `get_usage_policy`

### 要求

- 调用方身份；
- 权限 scope；
- 调用频率限制；
- 可见字段控制；
- 素材授权过滤；
- 全量审计；
- 不返回生产库字段；
- 不允许 MCP 绕过内容审核。

## 十一、首轮给 Codex 的执行指令

将下面内容直接作为第一次开发任务：

```text
请先阅读 README.md、docs/PRD.md 和 docs/CODEX_PLAN.md。

本轮只执行 Phase 0，不要开始图片生成、视频生成、自动发布、公共 MCP，也不要在没有确认字段的情况下连接生产数据库。

请完成：
1. 检查当前仓库结构和已有代码；
2. 在 docs/architecture/SYSTEM_CONTEXT.md 中记录真实技术栈、运行方式、服务边界和待确认项；
3. 在 docs/architecture/DATA_MAPPING.md 中建立生产数据到 AI 资产库的字段映射模板，包括允许字段、敏感字段、脱敏规则和授权范围；
4. 给出推荐的项目目录结构和数据库技术选择，但不要为了理想架构重写已有系统；
5. 建立 .env.example，禁止写入真实密钥；
6. 建立最小开发说明和本地启动方式；
7. 将不确定信息明确列为 TODO；
8. 完成后总结改动、风险和 Phase 1 的具体实施步骤。

安全要求：AI 服务不得拥有生产数据库写权限；不得提交真实隐私数据、生产凭据或真实用户联系方式。
```

## 十二、后续协作约定

- 产品需求变化优先更新 `docs/PRD.md`；
- 开发顺序变化更新本文件；
- 重要架构决定写入 `docs/architecture/ADR-xxxx-*.md`；
- 每个可独立验收的工作建立 GitHub issue；
- Codex 完成一个阶段后，应在 PR 中说明：完成内容、未完成内容、测试方式、数据库变更、风险与回滚方式；
- ChatGPT 后续可读取本仓库与 PR，继续补充 PRD、审查实施偏差并为 Codex 生成下一阶段任务。
