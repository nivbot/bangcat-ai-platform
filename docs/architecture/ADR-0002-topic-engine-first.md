# ADR-0002：采用选题引擎优先的内容架构

- 状态：Accepted
- 日期：2026-07-21

## 背景

原始规划把猫咪资产同步、角色卡和多平台文案生成作为主要内容闭环。该方案能够生成结构正确的内容，但容易形成稳定、同质、缺乏传播动机的猫咪介绍。

运营经验表明，内容表现首先取决于选题、钩子、平台形式和时机，其次才是文案质量。爆款案例具有参考价值，但直接复制标题、正文或镜头结构会产生版权、平台、品牌和长期同质化风险。

## 决策

将“选题引擎”设为猫咪资产库与内容生成器之间的核心领域层。

系统采用以下分层：

```text
趋势与参考案例
      ↓
案例抽象与模式库
      ↓
选题引擎
      ↓
猫咪资产与角色卡
      ↓
内容包生成
      ↓
原创、事实与授权检查
      ↓
人工审核和发布
      ↓
表现数据回流
```

具体决策：

1. 生成器不得默认直接读取第三方完整作品；
2. 参考案例先转为抽象模式，再参与候选生成；
3. 每个选题候选必须包含猫咪独有事实、当下信号和平台原生形式；
4. 选题评分必须可解释并保留权重版本；
5. 高相似度、高版权风险和高事实风险使用硬阻断；
6. MVP 先采用人工案例录入和规则评分；
7. 自动趋势采集在人工工作流验证有效后再实施；
8. 发布表现必须回流到选题、模式和评分版本。

## 后果

### 正面

- 避免“资料输入后直接写作文”的低价值流水线；
- 将传播经验沉淀为可复用资产；
- 可以解释为什么推荐某只猫和某个角度；
- 为图文、视频和未来 MCP 提供统一选题层；
- 能逐步使用真实表现修正判断；
- 更容易建立帮猫自己的系列栏目和内容 IP。

### 代价

- 数据模型和审核链路更复杂；
- 需要人工维护参考案例和模式；
- 相似度、版权和多模态分析增加成本；
- 内容表现归因不可能完全准确；
- 趋势连接器需要逐个平台评估合规性。

## 被否决的方案

### 方案 A：猫咪资料直接生成多平台内容

实现最简单，但选题弱、重复率高，无法沉淀传播方法。

### 方案 B：保存爆款全文并直接改写

短期可能提高相似度，但版权和品牌风险高，容易形成低质量换皮，无法成为长期产品能力。

### 方案 C：先建设全网自动抓取系统

工程和合规成本高，且尚未证明选题工作流本身有效。MVP 不采用。
