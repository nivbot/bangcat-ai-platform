# 帮猫 AI 内容资产库与 MCP 创作系统 PRD

- 文档状态：V0.1
- 当前阶段：MVP 规划
- 产品负责人：帮猫小本本
- 开发方式：Codex 辅助开发
- 协作仓库：`nivbot/ai_cat_create`

## 1. 项目背景

“帮猫小本本”现有小程序数据库中已经沉淀了大量真实猫咪资料，包括照片、基础信息、救助经历、领养状态、帖子、留言、助力记录和互动内容。

目前这些内容主要只能在自有小程序内被用户看到，外部传播能力有限。团队同时运营微信公众号、视频号、抖音、小红书等第三方账号，需要持续生产图文、短视频和系列化内容，以：

1. 增加待领养猫咪曝光，促进领养；
2. 宣传“帮猫小本本”平台与品牌；
3. 将真实猫咪沉淀为可持续创作的数字角色资产；
4. 建立“帮猫宇宙”等拟人化系列内容；
5. 未来通过 MCP 安全开放部分数字资产，让更多创作者参与；
6. 探索内容、授权、合作和创作收益反哺流浪猫的模式。

## 2. 产品定义

本项目不是一个可以任意操作业务系统的“全能 Agent”，而是一套受控、可追溯、可审核的 AI 内容生产系统。

系统由三层组成：

### 2.1 生产数据层

现有小程序业务数据库，继续承担真实业务运行。

约束：

- 仅允许专用同步服务以只读账号访问；
- AI 服务不得直连；
- 不允许 AI 结果反向自动写入生产库；
- 不允许向外部 MCP 暴露。

### 2.2 AI 内容资产层

独立数据库，用于保存适合 AI 创作的脱敏数据与衍生资产，包括：

- 猫咪基础创作资料；
- 可公开照片与媒体素材；
- 猫咪角色卡；
- 外貌特征与视觉一致性描述；
- 真实事实边界；
- 禁止虚构内容；
- 故事种子；
- 内容草稿；
- 审核记录；
- 模型调用日志与成本；
- 未来的角色形象、三视图、动作资产和视频素材。

### 2.3 MCP 服务层

面向内部 Agent、Codex 和未来授权创作者的受控工具层。

MCP 不直接开放数据库，而只提供有限、可授权、可审计的工具，例如：

- 搜索可创作猫咪；
- 获取猫咪公开角色卡；
- 获取经授权的媒体素材；
- 创建内容任务；
- 获取内容草稿；
- 提交外部作品；
- 查询授权与使用规则。

MVP 阶段先为 MCP 预留领域服务接口，不要求立即开放公共 MCP。

## 3. 产品目标

### 3.1 MVP 目标

跑通一条从真实猫咪资料到人工审核内容草稿的完整链路：

1. 从生产库只读同步猫咪资料；
2. 数据脱敏、清洗、结构化；
3. 自动或人工选择猫咪；
4. 生成并保存猫咪角色卡；
5. 选择内容方向；
6. 生成多平台文案与视频脚本草稿；
7. 自动检查事实、风险和格式；
8. 进入后台人工审核；
9. 审核通过后人工复制或下载发布；
10. 记录生成成本、耗时和反馈。

### 3.2 非目标

MVP 暂不包含：

- 自动发布到微信、抖音、小红书或视频号；
- 完整图片生成流水线；
- 完整视频生成流水线；
- 面向公众开放 MCP；
- 自动把 AI 内容写回生产数据库；
- 无人工审核的全自动运营；
- 复杂多 Agent 自主规划框架。

## 4. 用户角色

### 4.1 内容运营人员

- 查看系统推荐的猫咪；
- 创建内容任务；
- 选择平台、内容类型和调性；
- 审核、编辑、重写、通过或废弃草稿；
- 导出发布素材；
- 反馈内容质量。

### 4.2 管理员

- 配置生产库同步字段；
- 管理脱敏和公开规则；
- 配置模型供应商与预算；
- 管理提示词模板；
- 查看调用日志、错误、成本与审计记录；
- 管理未来 MCP 的权限与授权。

### 4.3 内部 Agent / Codex

- 通过受控服务读取 AI 资产；
- 创建和执行内容任务；
- 不得访问生产库凭据；
- 不得绕过权限和审核流程。

### 4.4 外部创作者（远期）

- 在授权范围内通过 MCP 获取可公开角色资料；
- 按规则创作内容；
- 提交作品和来源信息；
- 不得获取救助人、领养人、联系方式等隐私数据。

## 5. 核心业务流程

### 5.1 数据同步流程

1. 定时任务或管理员手动发起同步；
2. 同步服务使用生产库只读账号查询允许字段；
3. 对数据进行映射、脱敏、清洗和去重；
4. 写入 AI 数据库的原始镜像区；
5. 转换为统一猫咪资产结构；
6. 记录同步批次、来源版本、错误和更新时间；
7. 不对生产库执行任何写操作。

同步需支持：

- 首次全量同步；
- 按更新时间增量同步；
- 单只猫手动重新同步；
- 软删除或下架状态同步；
- 失败重试；
- 幂等执行。

### 5.2 选猫流程

系统支持两种方式：

- 人工选择指定猫咪；
- 系统按规则推荐。

MVP 推荐评分因素：

- 当前是否可领养；
- 资料完整度；
- 是否有足够可用照片；
- 距离上次曝光时间；
- 入住或待领养时长；
- 是否存在值得讲述的新事件；
- 内容历史是否重复；
- 管理员设置的优先级。

推荐结果必须展示原因，不允许只有黑盒分数。

### 5.3 角色卡生成流程

角色卡是后续所有内容创作的稳定基础，不等同于一次性文案。

角色卡至少包含：

- 名称与内部 ID；
- 当前领养状态；
- 品种、花色、性别、年龄等基础事实；
- 视觉识别特征；
- 性格标签及证据来源；
- 真实经历摘要；
- 适合的拟人化设定；
- 可持续故事主题；
- 口头禅或表达风格（可选）；
- 可公开事实；
- 不可公开信息；
- 禁止编造内容；
- 存疑信息；
- 角色卡版本和审核状态。

角色卡更新必须版本化，已发布内容应能追溯到当时使用的角色卡版本。

### 5.4 内容方向选择

MVP 提供以下内容类型：

1. 真实领养故事：以真实事实为主，促进领养；
2. 猫咪日常：从帖子、留言或真实事件提炼；
3. 温情短篇：允许文学化表达，但不得改变关键事实；
4. 搞笑拟人短篇：明确属于“帮猫小剧场”或虚构栏目；
5. 帮猫宇宙：角色持续发展、旅行、冒险或职业设定；
6. 平台宣传：自然介绍帮猫小本本、猫站或领养方式；
7. 短视频脚本：输出镜头、旁白、字幕、素材需求和时长。

系统必须标记内容的事实等级：

- `factual`：真实事实内容；
- `adapted`：基于真实事实的文学化改编；
- `fictional`：明确虚构的角色故事。

### 5.5 多平台生成

同一内容任务可以生成不同平台版本，但必须共享同一个故事核心和事实来源。

#### 公众号

- 相对完整的故事结构；
- 标题、导语、正文、小标题、结尾行动引导；
- 可附素材建议与排版建议。

#### 小红书

- 标题候选；
- 短段落正文；
- 更强的开头钩子；
- 适度表情符号；
- 话题标签建议；
- 避免虚假夸张和过度营销。

#### 抖音 / 视频号

- 视频标题；
- 15、30 或 60 秒脚本；
- 分镜；
- 旁白；
- 屏幕字幕；
- 所需照片、视频或后续 AI 素材；
- 封面文案；
- 发布说明。

### 5.6 自动质检

内容生成后必须进行独立质检，不应仅让同一次生成自行声明正确。

检查项包括：

- 猫咪名称、性别、年龄、状态是否与资产一致；
- 是否虚构疾病、救助经历、领养结果或人物关系；
- 是否泄露个人信息和联系方式；
- 是否把虚构故事包装成真实事件；
- 是否包含高风险医疗、领养或公益表述；
- 是否存在平台敏感词或明显低质量套话；
- 是否缺少领养状态和行动引导；
- 是否与近期内容高度重复；
- 输出格式是否完整；
- 字数和时长是否符合平台要求。

质检结果分为：

- 通过；
- 警告，可人工确认；
- 阻断，必须修改后重新检查。

### 5.7 人工审核台

审核页面需要同时展示：

- 猫咪真实资料；
- 来源照片和内容；
- 当前角色卡；
- 内容任务参数；
- 生成草稿；
- 事实引用；
- 质检结果；
- 模型与成本信息；
- 历史修改记录。

支持操作：

- 编辑；
- 局部重写；
- 整体重写；
- 切换平台版本；
- 通过；
- 驳回；
- 废弃；
- 复制或导出；
- 提交质量反馈。

## 6. 数据模型建议

具体字段以现有技术栈和生产库结构为准，Codex 在开发前应先完成数据库映射设计。

### 6.1 `source_sync_jobs`

同步批次、时间范围、状态、读取数量、写入数量、错误数量、错误摘要。

### 6.2 `source_cat_snapshots`

从生产库同步来的脱敏原始快照，保留来源 ID、来源更新时间、源数据哈希和同步批次。

### 6.3 `cat_assets`

统一后的猫咪 AI 资产主表，包含公开状态、领养状态、基础事实、资料完整度、最新角色卡版本。

### 6.4 `cat_media_assets`

猫咪照片与视频素材，包含来源、授权范围、可公开状态、质量标签、视觉角度和对象存储地址。

### 6.5 `cat_character_profiles`

版本化角色卡，包含结构化 JSON、提示词版本、模型信息、审核状态和生成依据。

### 6.6 `content_tasks`

内容任务，包含猫咪、内容类型、事实等级、目标平台、调性、时长、状态和发起人。

### 6.7 `content_drafts`

每个平台的草稿、结构化脚本、版本、审核状态、事实引用和最终采用内容。

### 6.8 `content_quality_checks`

质检结果、风险等级、问题列表、规则版本和模型结果。

### 6.9 `prompt_templates`

角色卡、选题、文案、视频脚本和质检提示词模板，必须版本化。

### 6.10 `ai_generation_runs`

模型供应商、模型名称、输入输出 token、费用、耗时、重试、错误和关联业务对象。

### 6.11 `audit_logs`

管理员、运营人员、Agent 和未来 MCP 调用的关键操作记录。

## 7. 模型与接口策略

### 7.1 MVP 必接

- 一个支持结构化输出的文本大模型接口；
- 自有生产库只读接口或同步服务；
- AI 数据库；
- 对象存储或现有媒体访问能力；
- 后台管理接口。

模型接入必须抽象供应商适配层，避免业务代码绑定单一模型。

统一能力接口建议：

- `generateStructuredObject()`
- `generateText()`
- `moderateOrCheck()`
- `estimateCost()`

### 7.2 MVP 暂不必接

- 图片生成 API；
- 视频生成 API；
- 自动剪辑 API；
- 第三方平台发布 API。

图文第一版优先使用真实猫照片和固定版式模板。视频第一版输出脚本、素材清单和人工剪辑建议。

### 7.3 成本控制

- 简单任务使用低成本模型；
- 高价值角色卡或复杂脚本再使用高质量模型；
- 输入前先压缩、去重和结构化资料；
- 缓存不变的角色卡与事实摘要；
- 重写尽量局部执行，不重复生成全部内容；
- 设置单任务、单日和单月预算；
- 保存 token 与费用；
- 支持模型降级和失败重试；
- 先以 10 只猫完成真实成本测试。

## 8. 安全、隐私与授权

必须在同步阶段完成字段白名单和脱敏，不应依赖提示词阻止模型泄露。

禁止进入 AI 资产库或 MCP 输出的内容包括：

- 救助人或领养人的手机号、微信号、地址等信息；
- 内部审核备注；
- 未公开的医疗信息；
- 内部运营账号和凭据；
- 生产数据库连接信息；
- 未获得公开授权的图片和视频；
- 任何可能定位个人或家庭的信息。

所有媒体资产需具备授权范围字段，例如：

- 仅内部使用；
- 帮猫官方渠道可用；
- 合作创作者可用；
- 公共 MCP 可用；
- 禁止 AI 二次生成。

## 9. MCP 预留设计

MVP 不急于对外开放，但领域服务应避免与后台页面强耦合，未来可包装为 MCP tools。

建议的首批 MCP 工具：

- `search_cats`
- `get_cat_profile`
- `get_cat_public_facts`
- `list_cat_media`
- `get_story_seeds`
- `create_content_task`
- `get_content_task`
- `submit_creator_work`
- `get_usage_policy`

每次调用必须包含调用方身份、授权范围、用途和审计记录。

外部 MCP 返回的信息应比内部后台更少，默认只提供审核通过的公开角色卡和授权素材。

## 10. 后台页面范围

### 10.1 仪表盘

- 今日待审核；
- 生成成功率；
- 本日与本月费用；
- 同步状态；
- 内容采用率；
- 风险告警。

### 10.2 猫咪资产列表

- 搜索和筛选；
- 资料完整度；
- 可创作状态；
- 角色卡状态；
- 最近曝光时间；
- 快速创建任务。

### 10.3 猫咪资产详情

- 同步资料；
- 媒体素材；
- 角色卡版本；
- 事实边界；
- 故事种子；
- 内容历史。

### 10.4 内容任务创建页

- 选择猫咪；
- 内容类型；
- 事实等级；
- 目标平台；
- 语气和长度；
- 补充指令；
- 是否生成多个方案。

### 10.5 审核台

- 原始资料与草稿并排；
- 质检问题定位；
- 编辑和重写；
- 审核流转；
- 导出。

### 10.6 系统设置

- 数据映射；
- 模型与密钥；
- 提示词模板；
- 预算；
- 用户权限；
- MCP 授权（远期）。

## 11. 状态机建议

### 11.1 内容任务

`pending -> preparing -> generating -> checking -> awaiting_review -> approved | rejected | failed | cancelled`

### 11.2 草稿

`draft -> checking -> needs_revision -> ready_for_review -> approved | rejected | archived`

### 11.3 角色卡

`missing -> generated -> needs_review -> approved -> superseded`

所有状态变更应记录操作者、时间和原因。

## 12. MVP 验收标准

使用至少 10 只资料质量不同的真实猫咪完成测试，并满足：

1. 生产库账号为只读，系统无法执行写操作；
2. 全量与增量同步可重复执行且不产生重复记录；
3. 敏感字段不会进入 AI 数据库；
4. 每只猫可生成结构化、可版本化的角色卡；
5. 至少支持公众号、小红书、视频号/抖音脚本三种输出；
6. 真实、改编、虚构内容有明确标识；
7. 草稿包含可追溯的事实来源；
8. 质检能阻断至少一组预设错误案例；
9. 审核人员可编辑、重写、通过、驳回和导出；
10. 每次模型调用可查看模型、token、费用、耗时和错误；
11. 单个任务失败不会影响其他任务；
12. 不存在自动发布和自动写回生产库的路径。

## 13. 效果指标

MVP 阶段重点不是粉丝增长，而是验证生产能力：

- 角色卡人工采用率；
- 草稿无需大改即可采用的比例；
- 单篇内容平均人工修改时间；
- 单只猫内容生成成本；
- 事实错误率；
- 内容重复率；
- 生成失败率；
- 从创建任务到可审核草稿的耗时。

进入运营验证后再观察：

- 发布频率；
- 阅读、完播、互动和关注；
- 猫咪详情页访问；
- 领养咨询和领养转化；
- 不同内容类型的效果差异。

## 14. 迭代路线

### Phase 0：技术摸底

- 盘点现有生产库结构和技术栈；
- 明确字段白名单与隐私规则；
- 建立环境和开发约定。

### Phase 1：数据隔离与资产库

- 建立 AI 数据库；
- 实现只读同步；
- 建立猫咪资产与媒体资产页面。

### Phase 2：文本内容闭环

- 角色卡生成；
- 内容任务；
- 多平台草稿；
- 质检；
- 人工审核台；

### Phase 3：模板化图文与视频脚本

- 图文模板；

- 视频脚本与素材清单；

## 15. 当前需要确认的技术信息

Codex 开始实现前，应在仓库中记录以下事实：

- 现有后台前后端技术栈；
- 生产数据库类型与版本；
- 猫咪、帖子、留言、媒体、领养状态对应表；
- 当前对象存储服务；
- 部署方式与环境；
- 现有鉴权与管理员体系；
- 计划使用的模型供应商；
- 可用于测试的脱敏数据；
- 当前代码是否会迁入本仓库，还是本仓库只做新服务。

这些信息缺失时，Codex 不应猜测生产数据库字段或直接编写不可逆迁移。
